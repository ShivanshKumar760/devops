Container Identity
GET /info

Fetch the container's hostname and internal IP. Inside Docker, the hostname is the short container ID — not your machine's hostname.

💡 Notice: The hostname is a short hash — that's the container ID. The IP (e.g. 172.18.x.x) is the container's address inside the Docker bridge network, not your machine's IP.
Internal DNS Communication INTERNAL
GET /talk-to-ping

api-service calls ping-service using just its container name as hostname. Docker's built-in DNS resolves "ping-service" → container IP automatically.

💡 How it works: api-service called http://ping-service:4000/pong — no IP address needed. Docker's embedded DNS server resolved "ping-service" to its container IP. This only works because both containers are on the same network (app-network).
External Internet Access EXTERNAL
GET /external

api-service calls a real public API (jsonplaceholder.typicode.com) from inside the container. Docker routes outbound traffic through the host's NAT.

💡 How it works: Docker uses NAT (Network Address Translation) — outbound traffic exits through the host machine's real IP. The external server sees your host's IP, not the container's internal IP (172.18.x.x).
Network Interfaces
GET /network

See all network interfaces inside the container. eth0 is the virtual NIC Docker creates — lo is the loopback interface (localhost inside the container).

💡 Notice: eth0 has a 172.x.x.x IP — that's the Docker virtual network. lo is 127.0.0.1 — localhost inside the container. Your host machine's IP (192.168.x.x) is NOT visible here.
Isolated Service ISOLATED
port 5000

isolated-service is on a separate Docker network. api-service CANNOT reach it internally — even though they're on the same host. This demonstrates network isolation.

💡 Network Isolation: isolated-service is on "isolated-network" while api-service is on "app-network". Docker treats them as separate virtual switches — no traffic can flow between them. This is how microservices enforce security boundaries.

// KEY CONCEPTS DEMONSTRATED

🔀 Bridge Network

The default Docker network type. Acts like a virtual switch. All containers on the same bridge can communicate using container names as hostnames.

🧭 Internal DNS

Docker runs a built-in DNS server. When a container calls "ping-service", Docker resolves it to the container's IP — no manual IP management needed.

🌍 NAT for Outbound

Containers reach the internet via the host's NAT. Outbound traffic exits through the host's IP. The external server never sees the container's 172.x.x.x address.

🚪 Port Mapping

host:container port mapping (e.g. 3000:3000) punches a hole in the firewall. Without it, the container is unreachable from your browser.

🔒 Network Isolation

Containers on different networks cannot communicate even on the same host. This is how Docker enforces security boundaries between services.

📦 Container Hostname

Each container gets a hostname equal to its short container ID. Other containers reference it by its service name, not the ID.
